# 基于 Qlib Code 模式的多因子策略开发工作流

这是一个基于 Qlib 的纯 Python 调用（Code 模式）的完整多因子策略开发工作流。该工作流具备生产级的持久化能力，并能处理真实的交易约束。

## 核心开发规范与深度自定义要求

### 1. 高性能数据转换与存储
- **目标**：指导如何将本地 CSV 数据（包含 OHLCV 及自定义因子）通过 Python API 转换为 Qlib 的 .bin 格式。
- **核心**：演示 `DumpBin` 工具类在代码中的调用逻辑，并解释 Feature 和 Label 的映射配置。

### 2. 代码驱动的任务管理 (Task/Workflow)
- **拒绝使用 YAML**：通过 Python 代码直接实例化并串联 `DataHandlerLP`、`DatasetH`、`Model` 及 `Trainer`。
- **持久化**：利用 Qlib 的 `Recorder` 和 MLflow 接口实现模型权重、预测结果及实验元数据的自动持久化。

### 3. 计算图因子扩展 (Expression Provider)
- **构建复杂因子**：利用 Qlib 的内置算子（Operators，如 `Ref`, `Mean`, `Std` 等）构建非线性的复杂因子。
- **特征集成**：说明如何将这些因子与 `Alpha158/360` 基础特征库进行集成。

### 4. 真实交易限制处理
- **回测配置**：必须包含对以下特殊交易限制的处理：
  - 停牌
  - 涨停（无法买入）
  - 跌停（无法卖出）
- **实现方式**：通过 `trade_exchange` 参数或自定义 `DealStock` 逻辑在 Strategy 和 Backtest 模块中实现这些约束。

### 5. 自动化寻优与全维度评价
- **超参数优化**：在代码流中集成超参数优化模块（如结合 Optuna）。
- **评价指标**：在回测结束后输出包含以下维度的评价：
  - RankIC
  - IR
  - 分层收益
  - 考虑了交易限制后的夏普比率和最大回撤

## 工具调用与准确性原则（核心）

### 强制约束
由于 Qlib 的以下部分接口繁杂且版本更迭快，必须确保代码的准确性：
- `DataHandler` 继承逻辑
- `Processor` 链式调用
- `Backtest` 中的 `trade_exchange` 配置项参数

### API 准确性核查流程
1. **查询优先**：遇到不确定的 Qlib 相关 API 调用、参数定义、或功能实现细节时，必须优先调用 **deepwiki MCP 工具** 搜索最新的官方文档或 GitHub 源码注释。
2. **版本适配**：通过查询确保代码示例的 API 签名、参数命名、和默认值与当前使用的 Qlib 版本完全兼容。

### 准确性要求
- 确保所提供的代码示例在 API 签名、参数名称（如 `limit_threshold` 或 `is_st` 等处理逻辑）上完全准确。
- 避免使用过时或不存在的 API。
