# Alpha-MQ Domain Context

Alpha-MQ is a quantitative multi-factor research and backtesting platform focusing on cross-sectional factor strategies.

## Language

**Symbol**:
A unique character code registered with an exchange to identify a specific stock, index, or ETF.
_Avoid_: Instrument, code, ticker, asset

**Factor**:
A raw financial predictive signal computed from historical price-volume or fundamental data.
_Avoid_: Feature, predictor

**Feature**:
A cleaned, normalized, or neutralized quantitative variable ready for direct input into a machine learning model.
_Avoid_: Factor, raw signal

**Label**:
The target variable for machine learning training, representing the forward return of a security over a specific holding period.
_Avoid_: Return, target, y-value

**Universe**:
The pre-defined set of symbols that a quantitative strategy or factor pipeline is restricted to select from.
_Avoid_: Target pool, stock pool, instruments, asset group

## Relationships

- A **Universe** consists of a pre-defined set of **Symbols** (e.g. CSI1000 constituents).
- A **Factor** is computed using historical data of a specific **Symbol** within a **Universe**.
- A **Feature** is generated by applying cleaning, scaling, or neutralization to a **Factor**.
- A **Label** represents the future holding-period return of a specific **Symbol** to serve as the target for learning from **Features**.

## Example dialogue

> **Dev**: "When we compute the **Factor** pool, do we fetch historical data of the entire market?"
> **Domain expert**: "No — the calculation should be strictly restricted to the **Symbols** within our specified **Universe** to avoid survival bias and reduce CPU overhead."
> **Dev**: "Understood. After getting raw **Factors**, do we immediately feed them into the model?"
> **Domain expert**: "No, raw **Factors** often have extreme values and collinearity. We must perform cross-sectional neutralization first to transform them into high-quality **Features** before feeding them to the model, aligned with their predictive **Labels**."

## Flagged ambiguities

- **Symbol vs Instrument**: Code and configurations used both terms ambiguously. Resolved: **Symbol** is the canonical domain term for exchange registration codes, keeping consistency with upstream GM and AKShare SDKs.
- **Factor vs Feature**: ML tasks often call everything a feature. Resolved: **Factor** refers to raw financial signals, while **Feature** is reserved for sanitized, model-ready variables.
- **Universe vs Instruments**: Configs used `instruments` for selection ranges. Resolved: **Universe** is the domain term for selection boundaries, while `instruments` is kept strictly as a backend Qlib config keyword.
- **Instrument Column in Data Schema**: To maintain seamless integration with Qlib's physical format specifications without performance overhead, DataFrame columns storing codes are named `"instrument"`. However, all business parameters, method signatures, logs, and variables in the Python codebase must strictly refer to them as **Symbol** or **Universe**.
